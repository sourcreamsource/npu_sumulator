## 🟢 1. 프로젝트 소개  

- **DeepLearning** 핵심 연산인 **Matrix Multiplication**의 가장 작은 단위인 `곱하고 더하기 (MAC)`를 NPU가 처리합니다.   

- AI 칩(NPU)의 핵심 연산인 **MAC(Multiply-Accumulate)** 을 순수 파이썬으로 직접 구현한 콘솔 시뮬레이터입니다.  

- **입력된 패턴**`(ex. image, OCR)`이 **십자가(Cross)** 인지 **X** 인지를 **필터**`(ex. weight)`와의 **유사도 점수**`(ex. 다음 뉴런)`로 판별합니다.  

- 3×3부터 25×25까지 다양한 크기를 지원하며, 크기별 **연산 시간**을 측정해 `시간 복잡도(BigO)`를 직접 확인합니다.

## 🟢 3. 구현 요약  

### 🟡 MAC 연산  

외부 라이브러리 없이 이중 반복문으로 직접 구현했습니다.  

```python
def mac(pattern, filter_):  
    score = 0.0  
    for i in range(len(pattern)):  
        for j in range(len(pattern[i])):  
            score += pattern[i][j] * filter_[i][j]  
    return score  
```

- 같은 위치의 값을 곱하고 모두 더한 결과가 **유사도 점수**입니다.  
- 점수가 높을수록 해당 필터의 패턴과 더 유사합니다.  
- 이 연산은 수학적으로 **내적(dot product)** 과 같은 구조입니다.  
- 패턴과 필터를 2차원 배열로 보더라도, 모든 칸을 일렬로 펼치면 두 벡터의 같은 위치 값을 곱해 더하는 계산이 됩니다.  
- 필터에서 1인 위치는 “이 위치에 값이 있으면 해당 모양일 가능성이 높다”는 의미를 갖습니다.  
- 입력 패턴도 같은 위치가 1이면 `1 × 1 = 1`이 되어 점수에 더해집니다.  
- 반대로 필터가 1인데 입력이 0이거나, 필터가 0인데 입력이 1이면 곱셈 결과가 0이라 점수에 기여하지 않습니다.  
- 따라서 입력 패턴과 필터의 1 위치가 많이 겹칠수록 내적 값이 커지고, 그 필터와 더 유사하다고 판단할 수 있습니다.  
- 예를 들어 Cross 입력과 Cross 필터는 1 위치가 대부분 겹쳐 점수가 높고, Cross 입력과 X 필터는 가운데 일부만 겹쳐 점수가 낮습니다.  

<br>

### 🟡 라벨 정규화  

data.json의 라벨 표기가 일관되지 않을 수 있어, 프로그램 내부에서 아래 규칙으로 통일합니다.  

| 원본 표기 | 표준 라벨 |  
|----------|----------|
| `'+'`, `'cross'` (대소문자 무관) | `Cross` |  
| `'x'` (대소문자 무관) | `X` |  

- 모든 비교와 출력은 표준 라벨(`Cross` / `X`)을 기준으로 수행합니다.  
- 라벨 정규화는 `normalize_label()` 함수로 분리했습니다.  
- 분리 이유는 expected 값과 filter 키가 서로 다른 표현을 쓰기 때문입니다.  
- 예를 들어 expected에는 `+`가 들어오고, filter 키에는 `cross`가 들어옵니다.  
- 이 둘을 매번 비교 위치에서 직접 처리하면 같은 조건문이 여러 곳에 반복됩니다.  
- 별도 함수로 분리하면 입력 라벨이 어디에서 오든 `Cross` 또는 `X`로 한 번에 통일할 수 있습니다.  
- 새로운 라벨이 추가될 때도 판정 로직 전체를 고치지 않고 정규화 함수와 필터 목록만 확장하면 됩니다.  
- 코드 구조 관점에서 필터 로드 단계와 expected 비교 단계가 모두 `normalize_label()` 하나에 의존합니다.  
- 따라서 라벨 규칙이 바뀌어도 수정 지점이 한 곳으로 제한됩니다.  
- 정규화 로직을 분리하지 않으면 `filters` 반복문, `patterns` 채점 반복문, 출력 비교 로직에 같은 변환 규칙이 중복됩니다.  
- 중복된 조건문은 한 곳만 수정하고 다른 곳을 놓치는 실수를 만들기 쉽습니다.  
- 현재 구조에서는 `normalize_label("cross")`, `normalize_label("+")`, `normalize_label("x")`가 항상 표준 라벨을 반환합니다.  
- 이후 로직은 원본 문자열을 신경 쓰지 않고 `Cross`와 `X`만 비교하면 됩니다.  
- 즉 정규화 함수는 입력 데이터의 표현 차이를 프로그램 내부 표준값으로 바꾸는 경계 역할을 하며, 유지보수성과 테스트 용이성을 높입니다.  

<br>

### 🟡 JSON 키 파싱 및 필터 선택 방식  

- `data.json`의 patterns 키는 `size_{N}_{idx}` 형식입니다.  
- 예를 들어 `size_13_3`은 `_` 기준으로 나누면 `["size", "13", "3"]`이 됩니다.  
- 프로그램은 `case_id.split("_")[1]` 값을 정수로 변환해 크기 `N`을 추출합니다.  
- 그 다음 `loaded_filters[N]`에서 같은 크기의 필터를 선택합니다.  
- 따라서 `size_5_1`은 5×5 필터, `size_13_3`은 13×13 필터, `size_25_2`는 25×25 필터와 비교됩니다.  
- 필터를 로드할 때도 `size_5`, `size_13`, `size_25` 키에서 같은 방식으로 N을 추출합니다.  
- 필터와 패턴의 크기가 맞지 않으면 해당 케이스만 FAIL로 처리하고 다음 케이스로 넘어갑니다.  
- 코드 레벨 흐름은 두 단계입니다.  
- 첫째, 필터 로드 단계에서 `for size_key, filter_dict in data.get("filters", {}).items()`로 `size_5`, `size_13`, `size_25`를 순회합니다.  
- 각 `size_key`에 대해 `n = int(size_key.split("_")[1])`을 실행해 5, 13, 25 같은 정수 크기를 얻습니다.  
- 그 다음 filter 키인 `cross`, `x`를 `normalize_label()`로 `Cross`, `X`로 바꿉니다.  
- 검증이 끝난 필터는 `loaded_filters[n] = {"Cross": cross_f, "X": x_f}` 형태로 저장합니다.  
- 즉 `loaded_filters`는 문자열 키가 아니라 정수 크기 N을 키로 갖는 딕셔너리입니다.  
- 둘째, 패턴 분석 단계에서 `case_id`인 `size_13_3`에서도 같은 방식으로 `n = int(case_id.split("_")[1])`을 구합니다.  
- 이후 `cross_f = loaded_filters[n]["Cross"]`, `x_f = loaded_filters[n]["X"]`로 같은 크기의 필터를 꺼냅니다.  
- 이 설계 덕분에 패턴 번호 idx가 무엇이든, 크기 N만 같으면 항상 올바른 `size_N` 필터와 매칭됩니다.

### 🟡 동점(epsilon) 처리 정책  

- 부동소수점 연산 특성상 두 점수가 수학적으로 동일해도 미세한 오차가 발생할 수 있습니다.  
- 컴퓨터는 2진수로 처리를 하는데  
- 컴퓨터는 64비트 같은 제한된 크기만 사용 → 무한 소수를 중간에 잘라서 저장  
- 정확한 값이 아니라 “가장 가까운 값(근사값)”으로 저장한다...  
- 두 값의 차이가 1e-9보다 작으면, 같다고 간주하자   

```python
EPSILON = 1e-9  

if abs(score_cross - score_x) < EPSILON:  
    result = "UNDECIDED"  
elif score_cross > score_x:  
    result = "Cross"  
else:  
    result = "X"  
```

두 점수의 차이가 `1e-9` 미만이면 **UNDECIDED(판정 불가)** 로 처리합니다.

### 🟡 성능 분석 (크기별 평균 연산 시간)  
- 각 크기별로 MAC 연산을 10회 반복 측정한 평균값입니다.  
- I/O 시간을 제외하고 연산 함수 호출 구간만 측정했습니다.  

- 코드에서 시간 측정 경계는 `measure()` 함수 내부에 있습니다.  
- 입력 받기, JSON 읽기, 필터 선택, 표 출력은 `measure()` 밖에서 수행됩니다.  
- `measure()` 안에서는 `t0 = time.perf_counter()`를 호출한 직후 `mac(pattern, filter_2d)`만 실행합니다.  
- 그리고 `t1 = time.perf_counter()`로 종료 시각을 기록합니다.  
- 즉 측정 구간은 `t0`와 `t1` 사이이며, 이 사이에 들어 있는 핵심 작업은 MAC 함수 호출뿐입니다.  
- 이 구조 때문에 콘솔 입력 시간, 파일 로드 시간, 결과 출력 시간은 평균 시간에 포함되지 않습니다.  
- 동일한 MAC 호출을 10회 반복하고 `(total / repeat) * 1000`으로 ms 단위 평균을 계산합니다.  

| 크기(N×N) | 평균 시간(ms) | 연산 횟수(N²) |  
|----------|-------------|-------------|
| 3×3      | 0.0013      | 9           |  
| 5×5      | 0.0029      | 25          |  
| 13×13    | 0.0144      | 169         |  
| 25×25    | 0.0496      | 625         |  

#### 시간 복잡도 분석  
- MAC 연산은 N×N 크기의 배열을 이중 반복문으로 처리합니다.  
- 각 원소를 한 번씩 곱하고 더하므로 연산 횟수는 정확히 **N²** 번입니다.  

- N이 2배 증가하면 연산 횟수는 4배 증가합니다.  
- 3×3(9회) → 5×5(25회) → 13×13(169회) → 25×25(625회) 순으로 급격히 늘어납니다.  
- 이를 **시간 복잡도 O(N²)** 이라고 합니다.  

실제 AI 모델은 수백×수백 크기의 필터를 수천 개 사용하므로, 이를 GPU/NPU로 병렬 처리하는 것이 필수적입니다.  

- 측정 결과도 연산 횟수가 늘어날수록 시간이 함께 증가하는 방향을 보입니다.  
- 3×3은 9번의 MAC 연산을 수행하고 평균 시간이 약 0.0013ms입니다.  
- 5×5는 25번의 MAC 연산을 수행하므로 3×3보다 연산량이 약 2.78배 많고, 측정 시간도 0.0029ms로 증가합니다.  
- 13×13은 169번의 MAC 연산을 수행하므로 5×5보다 연산량이 약 6.76배 많고, 측정 시간도 0.0144ms로 더 커집니다.  
- 25×25는 625번의 MAC 연산을 수행하므로 13×13보다 연산량이 약 3.70배 많고, 측정 시간도 0.0496ms까지 증가합니다.  
- 측정 시간 비율이 연산 횟수 비율과 완전히 같지는 않습니다.  
- 그 이유는 Python 반복문 오버헤드, 함수 호출 비용, 운영체제 스케줄링, CPU 캐시 상태 같은 잡음이 있기 때문입니다.  
- 하지만 크기가 커질수록 `N²` 연산 횟수 증가에 따라 평균 시간이 일관되게 증가하므로 O(N²) 경향을 확인할 수 있습니다.

### 🟡 FAIL 케이스 원인 분석  
- 현재 `data.json` 기준 실행 결과는 FAIL 2개입니다.  
  - `expected` 값의 `+`는 `Cross`로, `x`는 `X`로 정규화한 뒤 비교했기 때문에 라벨 표기 차이로 인한 실패가 발생하지 않았습니다.  
  - 필터 키도 `cross`와 `x`를 내부 표준 라벨인 `Cross`, `X`로 변환해 사용했습니다.  
  - 점수 비교는 `abs(score_cross - score_x) < 1e-9`이면 `UNDECIDED`로 처리하는 epsilon 정책을 적용했습니다.  

#### 첫 번째 FAIL 케이스는 `size_5_4`입니다.  
`size_5_4`의 입력 패턴은 Cross 모양이므로 Cross 점수는 9, X 점수는 1입니다.  
하지만 `expected` 값이 의도적으로 `x`로 들어 있어 판정 결과 `Cross`와 기대값 `X`가 달라 FAIL이 발생합니다.  
즉 이 실패는 MAC 로직 문제가 아니라 데이터 라벨이 의도적으로 틀린 케이스입니다.  

#### 두 번째 FAIL 케이스는 `size_13_3`입니다.  
`size_13_3`은 모든 값이 1인 13×13 패턴이라 Cross 필터 점수와 X 필터 점수가 모두 25입니다.  
두 점수 차이가 epsilon보다 작으므로 프로그램은 `UNDECIDED`로 판정합니다.  

`UNDECIDED`는 expected 라벨인 `Cross`와 일치하지 않으므로 FAIL로 처리됩니다.  
나머지 데이터의 각 패턴은 Cross 점수와 X 점수 차이가 충분히 커서 부동소수점 오차로 인한 오판정이 발생하지 않았습니다.  
또한 각 패턴의 크기와 해당 필터 크기가 일치해 크기 불일치 FAIL은 발생하지 않았습니다.

## 🟢 5. 심층 인터뷰 답변  

### 🟡 새로운 라벨을 추가한다면?   
- `o` 라벨을 추가한다면 `size_5` 안에 `"o": [[...]]` 형태의 필터가 필요합니다.  
- 그 다음 `normalize_label()`에 `o`를 표준 라벨 `O`로 바꾸는 규칙을 추가합니다.  
- 그리고 현재 `judge()`는 두 점수만 비교하므로, 여러 라벨 점수 중 가장 큰 값을 고르는 방식으로 확장해야 합니다.  
- 즉 확장 순서는 데이터 스키마 추가 → 라벨 정규화 추가 → 점수 계산 반복 구조 변경 → PASS/FAIL 비교 기준 변경입니다.  


<br>

### 🟡 1000×1000 패턴을 처리한다면?  
- 1000×1000 패턴은 원소가 1,000,000개입니다. 필터 하나와 비교할 때만 MAC 곱셈/덧셈이 1,000,000번 필요합니다.  
- flatten을 할 필요가 있으며, 메모리 관점에서는 1000×1000 배열을 여러 개 복사하지 않고, 가능한 한 같은 배열을 재사용하는 것이 중요합니다.  
- 실제 서비스 수준에서는 NumPy, GPU/NPU 같은 벡터화 또는 병렬 처리 기반 구현이 필요합니다.  

<br>

### 🟡 CPU와 NPU 관점의 병목현상과 이를 대처하는 방법은?  
- CPU는 한 번에 처리할 수 있는 데이터 수가 적음  
- NPU도 데이터를 연산 유닛으로 가져오는 메모리 대역폭이 부족하면 병목이 생깁니다.  

- 결국 AI 연산 최적화는 MAC 연산량을 병렬화하는 것과 데이터를 효율적으로 공급하는 것 두 가지가 핵심입니다.  

<br>

### 🟡 동점 처리 규칙을 "mode1 vs mode2"로 다르게 둔 이유를 요구사항 관점에서 설명할 수 있는가?  
- mode 1
